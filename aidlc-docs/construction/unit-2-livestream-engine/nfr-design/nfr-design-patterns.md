## 1. SignalR Group Management Pattern

**Mục đích**: Quản lý room membership và broadcast hiệu quả — mỗi viewer join room = join SignalR group.

```
Viewer joins room (GET /api/v1/livestream/rooms/{roomId}/join)
        │
        ├── Validate: room active + viewer eligible (BR-LS-03)
        ├── DB: INSERT viewer_sessions (RoomId, UserId, JoinedAt)
        ├── Redis: INCR viewer_count:{roomId}
        └── Return { roomId, agoraToken (nếu cần), signalREndpoint }

Client connects to LivestreamHub
        │
        └── OnConnectedAsync:
                ├── Validate JWT (hub auth)
                ├── Extract roomId từ query string
                ├── Groups.AddToGroupAsync(connectionId, $"room:{roomId}")
                └── Broadcast ViewerJoined event to group
                    { viewerCount: await GetViewerCountAsync(roomId) }

Viewer leaves room (disconnect hoặc explicit leave)
        │
        └── OnDisconnectedAsync:
                ├── Groups.RemoveFromGroupAsync(connectionId, $"room:{roomId}")
                ├── DB: UPDATE viewer_sessions SET LeftAt = now
                ├── Redis: DECR viewer_count:{roomId}
                └── Broadcast ViewerLeft event to group
                    { viewerCount: await GetViewerCountAsync(roomId) }
```

**Group naming convention**:
```
room:{roomId}           # Tất cả viewers + host trong 1 room
direct:{userId}         # DirectChat — mỗi user có 1 personal group
call:{sessionId}        # Private call signaling (host + viewer)
```

**Connection tracking** (in-memory per ECS task, không persist):
```csharp
// IConnectionTracker — track connectionId → userId mapping
// Cần để biết user nào đang connected khi broadcast targeted events
interface IConnectionTracker {
    void Add(string connectionId, Guid userId, Guid? roomId);
    void Remove(string connectionId);
    IEnumerable<string> GetConnectionIds(Guid userId);
    int GetConnectionCount(); // For CloudWatch metric
}
// Implementation: ConcurrentDictionary (thread-safe, in-memory)
```

**Redis backplane**: Khi có nhiều ECS tasks, SignalR Redis backplane đảm bảo broadcast đến đúng task chứa connection.

## 2. Viewer Count Cache Pattern

**Mục đích**: Real-time viewer count với Redis INCR/DECR atomic — không race condition, không DB load.

```
Join event:
    Redis INCR viewer_count:{roomId}
    → Broadcast updated count to room group (mỗi 5 giây hoặc on-change)

Leave event:
    Redis DECR viewer_count:{roomId}
    → Ensure count không xuống dưới 0: DECR + check, nếu < 0 → SET 0

Periodic sync (Background service mỗi 5 giây):
    → Read viewer_count:{roomId} từ Redis
    → Broadcast ViewerCountUpdated to room group
    → (Không sync về DB — DB chỉ dùng ViewerSessions để tính chính xác)

Redis restart recovery (lazy resync):
    GET viewer_count:{roomId}
        │
        ├── HIT  → return value
        └── MISS → SELECT COUNT(*) FROM viewer_sessions
                        WHERE room_id = ? AND left_at IS NULL
                   → SET viewer_count:{roomId} {count} EX 3600
                   → return count
```

**Thundering herd protection**: Redis single-threaded → 1000 INCR/giây xử lý tuần tự, không conflict. Không cần Lua script cho MVP.

**Broadcast throttle**: Không broadcast mỗi INCR/DECR (quá nhiều events) — dùng background timer broadcast mỗi 5 giây.

```csharp
// ViewerCountBroadcastService — IHostedService
// Mỗi 5 giây: đọc tất cả active rooms → broadcast count
// Chỉ broadcast nếu count thay đổi so với lần trước (delta check)
```

## 3. Redis Streams Chat Pattern

**Mục đích**: Room chat messages — append-only, bounded memory, TTL cleanup.

```
Send message flow:
Client → ChatHub.SendRoomMessageAsync(roomId, text)
        │
        ├── Validate: user in room + not banned (BR-RC-01, BR-RC-03)
        ├── Rate limit: 3 messages/giây per user per room (in-memory sliding window)
        ├── Profanity filter (BR-RC-01)
        ├── Redis XADD room:{roomId}:chat MAXLEN ~ 1000 *
        │       userId {userId}
        │       displayName {displayName}
        │       text {filteredText}
        │       timestamp {unixMs}
        ├── Broadcast to room group via SignalR
        │       { messageId, userId, displayName, text, timestamp }
        └── Return ACK to sender

Message format in Redis Stream:
    Entry ID: {unixMs}-{seq}  (auto-generated by Redis)
    Fields:
        userId: "uuid"
        displayName: "string"
        text: "filtered text"
        timestamp: "unix_ms"
        type: "message" | "gift" | "system"
```

**Rate limiting** (in-memory sliding window per connection):
```csharp
// Per-user rate limit: 3 messages/giây
// Dùng SlidingWindowRateLimiter (System.Threading.RateLimiting)
// Key: userId + roomId (không dùng Redis để tránh overhead)
// Nếu exceed → ChatHub throw HubException("Rate limit exceeded")
```

**Gift events** cũng lưu vào Redis Stream (type="gift") để có history đầy đủ.

## 4. Billing Tick Reliability Pattern

**Mục đích**: Đảm bảo coin deduction chính xác, không duplicate, không mất — financial integrity.

```
Private call active:
    Hangfire recurring job ProcessBillingTick(sessionId, tickNumber)
    Schedule: mỗi 10 giây per active session
        │
        ├── Check: session still active? (Redis call_session:{sessionId})
        │       └── INACTIVE → skip (call đã end)
        │
        ├── Check: viewer balance >= cost_per_tick?
        │       └── INSUFFICIENT → End call (BR-PC-05)
        │
        ├── DB Transaction:
        │       ├── INSERT billing_ticks (session_id, tick_number, coins_charged, ...)
        │       │       ON CONFLICT (session_id, tick_number) DO NOTHING  ← idempotency
        │       ├── [IF inserted = 1] → UPDATE coin_wallets SET balance -= coins_charged
        │       └── COMMIT
        │
        ├── [SUCCESS] → Broadcast BalanceUpdated to viewer's personal group
        │
        └── [FAIL after 3 retries] → End call (EndedBy=System, reason=BillingError)
                ├── UPDATE call_sessions SET ended_at = now, ended_by = 'System'
                ├── Broadcast CallEnded to call:{sessionId} group
                └── Log incident

Idempotency guarantee:
    UNIQUE INDEX idx_billing_ticks_session_tick ON billing_ticks(call_session_id, tick_number)
    → Duplicate execution → INSERT DO NOTHING → balance không bị deduct 2 lần
```

**Coin balance warning** (BR-PC-04):
```
Sau mỗi billing tick:
    IF viewer.balance < 100 coins:
        Broadcast LowBalanceWarning to direct:{viewerId}
        { balance: current, threshold: 100 }
```

**Call session state** (Redis Hash):
```
call_session:{sessionId}:
    status: "active" | "ended"
    hostId: uuid
    viewerId: uuid
    startedAt: unix_ms
    costPerTick: int
    tickNumber: int (current)
TTL: 4 giờ (= Agora token TTL)
```

## 5. Agora Token Management Pattern

**Mục đích**: Server-side token generation, 4-hour TTL, revocation on call end.

```
Private call request flow:
Viewer → POST /api/v1/livestream/calls/request { hostId }
        │
        ├── Validate: 1 pending request max (BR-PC-01)
        ├── Validate: không bị block lẫn nhau
        ├── DB: INSERT call_requests (viewer_id, host_id, status=Pending)
        ├── Broadcast CallRequest to direct:{hostId}
        │       { requestId, viewerId, viewerName, viewerPhoto }
        └── Schedule timeout job (30 giây → auto-reject)

Host accepts:
Host → POST /api/v1/livestream/calls/{requestId}/accept
        │
        ├── DB: UPDATE call_requests SET status=Accepted
        ├── DB: INSERT call_sessions (host_id, viewer_id, started_at, cost_per_tick)
        ├── Generate Agora tokens (host + viewer):
        │       channelName: "call_{sessionId}"
        │       TTL: 4 giờ
        │       role: Publisher (cả 2)
        ├── Redis: SET call_session:{sessionId} {hash} EX 14400 (4h)
        ├── Schedule billing tick job (mỗi 10 giây)
        ├── Broadcast CallAccepted to direct:{viewerId}
        │       { sessionId, agoraToken, channelName, hostAgoraToken }
        └── Return { sessionId, agoraToken, channelName } to host

Token revocation on call end:
        ├── Agora API: kick all users from channel (server-side)
        ├── Redis: DEL call_session:{sessionId}
        └── Hangfire: cancel billing tick jobs for sessionId
```

**Agora quota monitoring** (Hangfire daily job):
```
CheckAgoraQuotaAsync():
    → GET Agora usage API (current month minutes)
    → Redis SET agora_quota:current_month {minutes} EX {seconds_to_month_end}
    → IF minutes >= 9000: FeatureFlags.Disable("private-call")
    → IF minutes >= 8000: AlertService.Send("Agora quota 80%")
    → IF minutes < 8000 AND flag disabled: FeatureFlags.Enable("private-call")
```

## 6. PostgreSQL Partition Maintenance Pattern

**Mục đích**: Đảm bảo partition luôn tồn tại trước khi INSERT — self-healing, zero operational overhead.

```
Startup (IHostedService):
    PartitionMaintenanceService.StartAsync():
        │
        └── For month in [current, +1, +2]:
                EnsurePartitionExistsAsync("direct_messages", month)
                EnsurePartitionExistsAsync("conversations", month)

EnsurePartitionExistsAsync(table, month):
    partitionName = "{table}_{yyyy}_{mm}"
    startDate = first day of month
    endDate = first day of next month
    
    EXECUTE:
        CREATE TABLE IF NOT EXISTS {partitionName}
        PARTITION OF {table}
        FOR VALUES FROM ('{startDate}') TO ('{endDate}')
    
    → Log: "Partition {partitionName} ensured"

Hangfire monthly job (ngày 25, 00:00 UTC):
    CreateNextMonthPartitionAsync():
        → EnsurePartitionExistsAsync("direct_messages", nextMonth)
        → EnsurePartitionExistsAsync("conversations", nextMonth)
        → Log: "Monthly partition creation completed"
```

**Query safeguard** (Repository pattern):
```csharp
// DirectMessageRepository — enforce SentAt filter
public async Task<List<DirectMessage>> GetMessagesAsync(
    Guid conversationId,
    DateTimeOffset from,      // Required — no default
    DateTimeOffset? to = null) {
    
    // 'from' is required — caller must provide SentAt range
    // Nếu cần "recent messages" → dùng GetRecentMessagesAsync() wrapper
    return await _db.DirectMessages
        .Where(m => m.ConversationId == conversationId
                 && m.SentAt >= from
                 && (to == null || m.SentAt < to))
        .OrderBy(m => m.SentAt)
        .ToListAsync();
}

// Convenience wrapper cho common case
public Task<List<DirectMessage>> GetRecentMessagesAsync(
    Guid conversationId, int days = 30) =>
    GetMessagesAsync(conversationId, DateTimeOffset.UtcNow.AddDays(-days));
```

## 7. DirectChat Delivery Pattern

**Mục đích**: 1-1 message delivery với < 500ms p95, push notification fallback khi offline.

```
Send DirectMessage:
Client → ChatHub.SendDirectMessageAsync(recipientId, text)
        │
        ├── Validate: conversation exists + not blocked (BR-DC-01, BR-DC-03)
        ├── Rate limit: 10 messages/phút per user (in-memory)
        ├── DB Transaction:
        │       ├── INSERT direct_messages (conversation_id, sender_id, content, sent_at)
        │       └── UPDATE conversations SET last_message_at = now, unread_count += 1
        │
        ├── Check: recipient online? (IConnectionTracker.GetConnectionIds(recipientId))
        │       │
        │       ├── ONLINE → SignalR send to direct:{recipientId}
        │       │       { messageId, senderId, senderName, text, sentAt }
        │       │       → < 500ms p95 target
        │       │
        │       └── OFFLINE → Push notification (BR-DC-04)
        │               → SNS/FCM: "Tin nhắn mới từ {senderName}"
        │               → (Future: implement in Unit 4)
        │
        └── Return ACK { messageId, sentAt }

Block flow:
POST /api/v1/direct-chat/conversations/{conversationId}/block
        │
        ├── DB: INSERT blocks (blocker_id, blocked_id)
        ├── DB: UPDATE conversations SET is_hidden_for_blocker = true
        │                                is_hidden_for_blocked = true
        └── SignalR: Broadcast ConversationHidden to both parties
```

**Conversation visibility** (BR-DC-03):
```
GetConversationsAsync(userId):
    → SELECT * FROM conversations
      WHERE (user1_id = userId OR user2_id = userId)
        AND is_hidden_for_{role} = false
    → Không trả về conversations bị ẩn do block
```

## 8. Feature Flag Pattern (Agora Quota)

**Mục đích**: Dynamic disable/enable features mà không cần restart — Redis-backed.

```
IFeatureFlagService:
    IsEnabledAsync("private-call"):
        → Redis GET feature_flag:private-call
        → "true" → enabled
        → "false" → disabled
        → NULL → enabled (default — fail-open)

    SetAsync("private-call", false):
        → Redis SET feature_flag:private-call "false"
        → (No TTL — persists until explicitly changed)

Usage in controller:
    POST /api/v1/livestream/calls/request:
        IF NOT await _featureFlags.IsEnabledAsync("private-call"):
            return 503 { error: "private_call_unavailable",
                         message: "Private call tạm thời không khả dụng" }
```

**Feature flags trong Unit 2**:
| Flag | Default | Disable trigger |
|---|---|---|
| `private-call` | enabled | Agora quota >= 9,000 phút |
| `public-stream` | enabled | (Manual only — extreme case) |
| `room-chat` | enabled | (Manual only) |

## 9. CloudWatch Custom Metrics Pattern

**Mục đích**: Proactive monitoring — scale và alert dựa trên business metrics, không chỉ CPU/Memory.

```
MetricsPublisherService (IHostedService, publish mỗi 30 giây):
        │
        ├── SignalR.ConnectionCount
        │       → IConnectionTracker.GetConnectionCount()
        │       → PutMetricData(Namespace="LivestreamApp/SignalR")
        │       → ECS Auto-Scale trigger: > 4,000 → scale out
        │
        ├── SignalR.ActiveRooms
        │       → Count of rooms with active SignalR groups
        │
        └── Agora.UsageMinutes
                → Redis GET agora_quota:current_month
                → PutMetricData(Namespace="LivestreamApp/Agora")
                → Alert: > 8,000 (warning), > 9,000 (critical)

BillingMetricsService (publish per billing tick):
        ├── Billing.TickLatency → actual execution time
        └── Billing.FailedTicks → increment on failure

CloudWatch Alarms:
    SignalR.ConnectionCount > 4,000 → ECS scale out
    Billing.FailedTicks > 0 → SNS alert to ops team
    Billing.TickLatency > 30s → SNS alert
    Agora.UsageMinutes > 8,000 → SNS alert (warning)
    Agora.UsageMinutes > 9,000 → SNS alert (critical) + auto-disable
```

## 10. S3 Chat Archive Pattern

**Mục đích**: Daily export room chat từ Redis Streams lên S3 — long-term archive, compliance.

```
ExportRoomChatToS3Job (Hangfire, daily 02:00 UTC):
        │
        ├── Load all rooms with active streams (Redis SCAN room:*:chat)
        └── For each room:
                ├── XRANGE room:{roomId}:chat - + (read all entries)
                ├── Transform to JSON Lines format
                ├── S3 PutObject:
                │       key: chat-archive/{year}/{month}/{roomId}/{date}.jsonl
                │       content-type: application/x-ndjson
                ├── [SUCCESS] → Log "Exported {count} messages for room {roomId}"
                └── [FAIL after 3 retries] → Hangfire Dead Letter Queue
                        → Admin retry từ Hangfire Dashboard

JSON Lines format (1 message per line):
    {"id":"1711029600000-0","userId":"uuid","displayName":"User","text":"Hello","timestamp":1711029600000,"type":"message"}
    {"id":"1711029610000-0","userId":"uuid","displayName":"User","text":"❤️","timestamp":1711029610000,"type":"gift","giftId":"heart"}
```

**S3 key structure**:
```
chat-archive/
  2026/
    03/
      {roomId}/
        2026-03-22.jsonl
        2026-03-23.jsonl
```
